# LLog-Analysis-and-Observability-Challenge-6

## 📌 Overview 

This project focuses on analyzing distributed system logs to understand system behavior, detect anomalies, and identify the root cause of a production incident.

Instead of adding new logs or instrumentation, the goal is to work with existing telemetry data and extract meaningful insights about system performance and failures.

The analysis simulates a real-world observability scenario where operations teams need to answer:

- When did the system degrade?
- Which services were affected?
- What patterns explain the incident?
- How does the incident compare to normal system behavior?

---

## 🎯 Objective

The main objective is to transform raw logs into a structured incident analysis by:

- Identifying the worst-performing time window
- Measuring system degradation using a bad rate metric
- Detecting impacted services and endpoints
- Comparing incident behavior against normal system operation (baseline)
- Supporting conclusions with reproducible data analysis

---

## 📊 Dataset Description

The dataset contains logs generated by a distributed system with multiple services.

Each record includes:

- `timestamp_event`: event occurrence time
- `received_at`: ingestion time
- `service_name`: service that generated the log
- `severity`: log level (INFO, WARN, ERROR, CRITICAL)
- `message`: log message description
- `method`: HTTP method
- `endpoint`: API endpoint
- `status_code`: HTTP response code
- `latency_ms`: request latency
- `trace_id`: distributed trace identifier
- Additional metadata (host, region, env, log_type)

---

## ⚙️ Methodology

### 1. Data ingestion
The dataset is loaded from a CSV file stored in Google Drive and structured using pandas.

### 2. Data preprocessing
- Timestamp conversion to datetime format
- Validation of schema integrity
- Null and missing value checks

### 3. Time-based aggregation
Logs are grouped into 5-minute windows to analyze system behavior over time.

### 4. Failure definition
A log is considered a failure if:
- severity is ERROR or CRITICAL
- OR status_code >= 500

This is used to compute the **bad rate**.

### 5. Incident detection
The worst time window is identified based on:
- highest bad rate
- minimum event threshold

### 6. Comparative analysis
Incident behavior is compared against baseline system behavior using:
- bad rate
- latency
- error rates

---

## 🚨 Key Findings

- A major incident occurred on **2026-01-10 11:10 UTC**
- During this window, the system reached a **bad rate of ~58%**
- The normal system baseline was around **14%**
- Latency increased significantly during the incident (~3x baseline)

### Affected services:
- orders-service (primary impact)
- inventory-service (secondary impact)

### Main failure patterns:
- inventory lock timeout
- database deadlocks
- cascading order failures

### Affected endpoints:
- /orders/create
- /orders/cancel
- /orders/status
- /inv/reserve

---

## 🧠 Conclusion

The incident was caused by a concurrency and resource contention issue in the inventory system, which propagated into the order processing flow.

The failure pattern suggests:
- high contention on shared resources
- database locking issues
- cascading failures across dependent services

Overall, this was not an isolated error, but a systemic degradation triggered by internal resource saturation.
